Day 19. Common UX Alignment

이번 단계는 새 기능을 추가하는 날이 아니라, 데모 모드와 실제 모드가 같은 제품처럼 보이도록 문구와 결과 영역 톤을 맞추는 날입니다. 왼쪽 액션 패널 설명, 중앙 결과 영역 안내, 입력창 예시 문구, 토스트 문장 리듬을 공통 UX 규칙으로 정리합니다.

Day 19의 핵심은 layout 재설계가 아니라 copy polishing과 interaction tone alignment입니다.

1. 오늘 수정하는 파일과 역할

action_panel.dart

왼쪽 주요 기능 패널입니다. 버튼 설명과 모드 전환 카드 설명을 공통 제품 톤으로 맞춥니다.

ready_state.dart

중앙 결과 영역입니다. 준비 상태 문구, 결과 요약과 후속 제안의 라벨 구조를 더 분명하게 정리합니다.

text_command_composer.dart

텍스트 명령 입력창입니다. placeholder와 버튼 문구를 더 친절하고 일관된 제품형 문장으로 바꿉니다.

home_page.dart, demo_home_page.dart

실제 모드와 데모 모드에서 보여주는 토스트와 결과 문장 리듬을 공통 규칙으로 맞춥니다.

기존 문구는 기능 설명 중심이었고 길이도 조금씩 달랐습니다. Day 19에서는 같은 구조의 문장으로 정리해서 버튼들이 한 세트처럼 읽히게 만듭니다.

3. ReadyState 결과 영역 정리

중앙 영역은 공통 UX에서 가장 크게 보이는 부분입니다. 준비 상태 설명을 더 제품형 문장으로 바꾸고, followUp에는 고정 라벨 다음 제안을 붙여 구조를 더 쉽게 이해하게 만듭니다.

Text(
  isBusy
      ? '요청을 분석하고 다음 작업을 준비하고 있습니다. 잠시만 기다려 주세요.'
      : '준비 완료. 음성 듣기, 화면 읽기, 텍스트 입력 중 원하는 방식으로 작업을 시작할 수 있습니다.',
  textAlign: TextAlign.center,
  style: TextStyle(
    fontSize: 14,
    height: 1.6,
    color: surfaceTheme.textMuted,
  ),
),
if (followUp != null) ...[
  const SizedBox(height: 14),
  Text(
    '다음 제안',
    style: TextStyle(
      fontSize: 12,
      fontWeight: FontWeight.w700,
      color: surfaceTheme.textMuted,
    ),
  ),
  const SizedBox(height: 8),
  Container(
    width: double.infinity,
    padding: const EdgeInsets.all(14),
    decoration: BoxDecoration(
      color: surfaceTheme.contentBackground,
      borderRadius: BorderRadius.circular(12),
      border: Border.all(color: surfaceTheme.border),
    ),
    child: Text(followUp!),
  ),
],

4. 입력창 문구 정리

입력창은 사용자가 가장 바로 상호작용하는 곳이라 문구가 중요합니다. 모호한 설명형 placeholder 대신 바로 따라 쓸 수 있는 예시를 주는 편이 더 친절합니다.

decoration: const InputDecoration(
  hintText: '예: 유튜브에서 고양이 영상 찾아줘',
  border: OutlineInputBorder(),
),
ElevatedButton(
  onPressed: widget.isBusy ? null : _submit,
  child: Text(widget.isBusy ? '요청 준비 중...' : '텍스트 명령 실행'),
),

5. 실제 모드와 데모 모드 토스트 톤 맞추기

두 모드가 다른 앱처럼 보이지 않으려면 같은 상태에서 비슷한 문장 리듬을 써야 합니다. Day 19에서는 listening, processing, success 토스트 문구를 같은 패턴으로 정리합니다.

Listening

'음성을 듣고 있습니다. 요청을 받는 중입니다.'

Processing

'요청을 분석하고 다음 작업을 준비하고 있습니다.'

Success

'결과를 준비했습니다. 이어서 다음 작업도 안내할 수 있습니다.'

실제 모드의 성공 토스트는 followUp가 있으면 그 문구를 먼저 보여주고, 없을 때만 공통 성공 문장을 보여주도록 유지합니다.

void _showCommandResultToast(CommandResponseModel response) {
  switch (response.status) {
    case CommandResponseStatus.success:
      showAppToast(
        context,
        response.followUp ?? '결과를 준비했습니다. 이어서 다음 작업도 안내할 수 있습니다.',
        title: '작업 완료',
        state: AppToastState.success,
      );
      break;
    case CommandResponseStatus.warning:
      showAppToast(
        context,
        response.summary,
        title: '확인 필요',
        state: AppToastState.warning,
      );
      break;
    case CommandResponseStatus.error:
      showAppToast(
        context,
        response.summary,
        title: '작업 실패',
        state: AppToastState.error,
      );
      break;
  }
}

6. 데모 시나리오 문구도 같은 제품 톤으로 정리

데모 응답이 너무 “가짜 예시”처럼 보이면 실제 모드와 거리가 생깁니다. 그래서 시나리오 summary와 followUp도 조금 더 제품형 문장으로 다듬습니다.

case DemoScenario.youtube:
  return const _DemoResult(
    summary: '유튜브 검색 결과를 정리했습니다. 요청한 키워드와 관련된 대표 영상을 바로 확인할 수 있습니다.',
    followUp: '첫 번째 영상을 바로 재생할까요?',
  );

case DemoScenario.naverMap:
  return const _DemoResult(
    summary: '지도 검색 결과를 정리했습니다. 현재 위치 기준으로 가장 관련성 높은 장소를 확인했습니다.',
    followUp: '길찾기 안내를 시작할까요?',
  );

case DemoScenario.secureInput:
  return const _DemoResult(
    summary: '보안 입력이 필요한 화면으로 판단했습니다. 민감한 정보 보호를 위해 일부 동작은 제한됩니다.',
    followUp: '보안 입력 모드를 유지한 채 계속 진행할까요?',
  );

7. 오늘 꼭 이해해야 하는 것

  • Day 19는 새 기능보다 공통 UX의 인상과 리듬을 맞추는 단계입니다.
  • 데모와 실제 모드가 완전히 같은 문장을 쓸 필요는 없지만, 같은 규칙의 문장을 써야 같은 제품처럼 느껴집니다.
  • ReadyState의 라벨 구조를 고정하면 사용자가 결과를 더 빠르게 읽습니다.
  • 토스트 문구는 짧아도 상태 리듬이 같아야 제품 톤이 안정됩니다.

8. Day 19 완료 기준

  • ActionPanel 설명 문구가 더 통일감 있게 정리된다.
  • ReadyState 준비 상태와 후속 제안 라벨이 더 명확해진다.
  • 입력창 placeholder가 더 직접적이고 친절해진다.
  • 실제 모드와 데모 모드 토스트 문구 구조가 거의 같은 리듬을 가진다.
  • 두 모드가 같은 제품군으로 느껴지는 공통 UX 완성도가 높아진다.
