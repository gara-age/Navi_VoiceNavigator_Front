Day 24

Web Automation Recovery Flow

이 문서는 웹 자동화 JSON plan 실행이 실패했을 때, 후보 축약 정보를 기반으로 Agent에게 다시 계획을 요청하고 1회 재실행하는 recovery 흐름을 정리한 튜토리얼입니다. 기본 실행 성공 시에는 추가 토큰 비용이 들지 않고, 실패 시에만 보조 프롬프트가 발생하도록 설계되어 있습니다.

핵심 원칙: recovery는 모든 에러에 대해 수행하지 않습니다. 오직 target 해석 실패처럼 “선택 문제”에 대해서만 실행하고, 재시도는 1회만 허용합니다.

1. 전체 recovery 흐름

LLM plan 생성
  -> Flutter가 JSON plan 실행
  -> Python/Playwright step 실행
  -> target 선택 실패
  -> Python이 recovery payload 반환
  -> Flutter가 후보 축약과 실패 step을 포함한 recovery prompt 생성
  -> Agent에게 수정된 plan 재요청
  -> 새 plan 1회 재실행

즉 정상 경로는 기존과 동일하고, 실패 시에만 recovery branch가 열립니다.

3. TargetResolver는 reason_code를 반환해야 한다

local_server/app/automation/browser/target_resolver.py

recovery는 “무슨 종류의 실패였는가”를 알아야 하므로, 문자열 오류만 던지지 말고 표준화된 reason_code를 같이 실어야 합니다.

class TargetResolutionError(RuntimeError):
    def __init__(
        self,
        message: str,
        reason_code: str = "target_not_found",
        debug_snapshot: str = "",
    ) -> None:
        super().__init__(message)
        self.reason_code = reason_code
        self.debug_snapshot = debug_snapshot

현재 구현에서는 우선 target_not_found를 기본 reason code로 사용합니다.

4. candidate_inventory는 recovery용 후보 축약을 만들어야 한다

local_server/app/automation/browser/candidate_inventory.py

이미 들어간 inventory와 scoring을 그대로 재사용하되, LLM에 직접 보내기 좋은 작은 요약을 만들어야 합니다.

def build_recovery_candidate_summary(
    page: Any,
    target: dict[str, Any],
    limit: int = 5,
) -> dict[str, Any]:
    inventory = build_candidate_inventory(page)
    ranked = select_inventory_candidates(page, target)
    top_candidates = [
        compact_candidate(item.get("candidate") or {})
        for item in ranked[:limit]
        if item.get("candidate")
    ]
    return {
        "intent": classify_candidate_intent(target),
        "description": target.get("description"),
        "top_candidates": top_candidates,
        "summary": inventory.get("summary") or {},
    }

이 구조의 장점은 전체 HTML이나 전체 후보를 다시 보내지 않고도, recovery에 필요한 top candidates만 아주 작게 보낼 수 있다는 점입니다.

최신 안정화: 현재는 top_candidates만 보내는 것이 아니라, llm_summary 안에 top_matches, context_buckets, relation_summary를 함께 넣어서 LLM이 페이지 맥락을 더 잘 읽을 수 있도록 보강했습니다.

5. CommandExecutor는 structured failure를 반환해야 한다

local_server/app/automation/browser/command_executor.py

step 실행 실패 시 단순 문자열만 돌려주면 Flutter에서 recovery를 orchestrate하기 어렵습니다. 실패한 step과 recovery payload를 같이 반환해야 합니다.

except Exception as exc:
    failure = self._build_step_error(
        page,
        step,
        exc,
        completed_steps=completed_steps,
        observations=observations,
    )
    completed_steps.append(
        {
            "step": step_no,
            "action": action,
            "status": "error",
            "detail": str(exc),
        }
    )
    failure["steps"] = completed_steps
    failure["observations"] = observations
    return failure
def _build_step_error(...):
    failure = {
        "status": "error",
        "reason": reason,
        "failed_step": step,
        "steps": completed_steps,
        "observations": observations,
    }
    if self._is_recoverable_target_error(exc):
        failure["reason"] = target_exc.reason_code
        failure["recovery"] = {
            "reason_code": target_exc.reason_code,
            "debug_snapshot": target_exc.debug_snapshot,
            "candidate_summary": build_recovery_candidate_summary(
                page,
                step.get("target") or {},
            ),
        }
        failure["detail"] = reason
    return failure

7. Dart 결과 모델은 recoverable failure를 판별할 수 있어야 한다

lib/shared/services/json_automation_runner_service.dart
bool get isRecoverableFailure {
  if (success) {
    return false;
  }
  final recovery = recoveryPayload;
  final reasonCode = recovery?['reason_code']?.toString().trim();
  return recovery != null &&
      reasonCode != null &&
      reasonCode.isNotEmpty &&
      const {
        'target_not_found',
        'no_visible_candidate',
        'candidate_blocked',
        'ambiguous_target',
      }.contains(reasonCode);
}

이렇게 해두면 SessionController는 raw JSON parsing을 직접 하지 않고도 recovery 가능 여부를 빠르게 판단할 수 있습니다.

8. SessionController가 recovery orchestration의 중심이다

lib/features/session/application/session_controller.dart

이 파일에서 plan 실행 실패를 감지하고, recovery prompt를 만들어 Agent에게 다시 보냅니다.

var execution = await JsonAutomationRunnerService.instance.runPlan(
  rawPlan: response.body,
);
if (!execution.success && execution.isRecoverableFailure) {
  execution = await _attemptAutomationRecovery(
    baseUri: uri,
    originalCommand: command,
    originalPlan: response.body,
    execution: execution,
  );
}
Future<JsonAutomationRunnerResult> _attemptAutomationRecovery({
  required Uri baseUri,
  required String originalCommand,
  required String originalPlan,
  required JsonAutomationRunnerResult execution,
}) async {
  if (_maxAutomationRecoveryAttempts < 1) {
    return execution;
  }

  final recoveryPrompt = _buildAutomationRecoveryPrompt(
    originalCommand: originalCommand,
    originalPlan: originalPlan,
    execution: execution,
  );
  if (recoveryPrompt == null) {
    return execution;
  }

  final recoveredPlan = await _requestAutomationRecoveryPlan(
    baseUri: baseUri,
    recoveryPrompt: recoveryPrompt,
  );
  if (recoveredPlan == null) {
    return execution;
  }

  return JsonAutomationRunnerService.instance.runPlan(
    rawPlan: recoveredPlan,
  );
}

9. recovery prompt는 전체 HTML이 아니라 실패 step과 후보 축약만 사용한다

사용자 원래 요청:
$originalCommand

$reasonCode

중요: recovery에서는 step patch만 받는 구조도 가능하지만, 현재 구현은 안정성을 위해 수정된 전체 plan JSON을 다시 받는 방식입니다. 대신 후보 요약은 이제 llm_summary를 우선 사용하므로 토큰은 작게 유지하면서도 맥락은 더 풍부하게 전달합니다.

10. 재시도는 1회만

lib/features/session/application/session_controller.dart
static const int _maxAutomationRecoveryAttempts = 1;

무한 재시도는 페이지 상태를 더 꼬이게 만들고 토큰 비용도 커집니다. 그래서 recovery는 반드시 1회로 제한하는 것이 좋습니다.

11. 구현 후 확인할 체크리스트

  • 정상 실행 성공 시에는 recovery 요청이 발생하지 않는가
  • target_not_found 실패 시 Python payload에 recovery가 들어가는가
  • Flutter가 execution.isRecoverableFailure를 true로 판단하는가
  • Recovery prompt가 JSON 후보 축약만 포함하고 전체 HTML은 포함하지 않는가
  • 재실행은 1회만 일어나는가

12. 다음 후순위 작업

  1. relation group 확장: search input + submit button, autocomplete input + suggestion list 같은 action group을 추가하면 plan 생성 품질이 더 좋아집니다.
  2. locator_hint 안정화: data-testid, aria attr, stable parent context를 더 활용하면 fallback selector 품질이 좋아집니다.
  3. step patch 모드로 경량화: 현재는 전체 plan을 다시 받습니다. 이후에는 실패 step만 교체하는 방식으로 토큰을 더 줄일 수 있습니다.
  4. recovery prompt 템플릿 세분화: intent가 button/input/tab/dialog인지에 따라 프롬프트 지시문을 더 구체화하면 재질문 정확도가 좋아집니다.
