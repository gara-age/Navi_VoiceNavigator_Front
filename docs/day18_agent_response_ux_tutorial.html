Day 18

Agent Response UX Finalized

18일차는 실제 모드에서 Agent 응답을 더 자연스럽게 보이도록 정리하는 단계입니다. 단순히 isError만 보는 대신, 응답 상태를 success / warning / error로 구분하고, follow-up이 끝났을 때 pending 상태를 자동 정리하도록 구조를 확장합니다.

핵심 목표: CommandResponseStatus 도입, completesFollowUp 도입, fallback 응답 정리, 그리고 실제 모드 토스트를 응답 상태와 직접 연결합니다.

오늘 수정할 파일

  • lib/shared/models/response_models.dart
  • lib/features/session/application/session_controller.dart
  • lib/features/home/presentation/home_page.dart

실제 변경 지점 정확히 보기

문서 설명용 범위와 실제 코드 변경 지점을 분리해서 보면 더 정확합니다. 현재 코드 기준으로 18일차 핵심 변경은 아래와 같습니다.

  • lib/shared/models/response_models.dart: CommandResponseStatus enum 추가, status 필드 추가, completesFollowUp 필드 추가
  • lib/features/session/application/session_controller.dart: fallback 응답에 status: 'error'와 completesFollowUp: true 추가, _resolveStatus() 추가, _applyResponse()에서 pending 자동 정리
  • lib/features/home/presentation/home_page.dart: 응답 상태를 success / warning / error 토스트로 분기하는 소비 지점

왜 이 작업이 필요한가

  • isError 하나만으로는 경고 상태를 표현하기 어렵습니다.
  • follow-up이 끝났는데 pendingAction과 pendingTarget이 남아 있으면 다음 짧은 입력이 잘못 해석될 수 있습니다.
  • HTTP 연결 실패, API 주소 미설정, 정상 처리 완료를 같은 규칙으로 UI에 전달하려면 상태 모델이 더 명확해야 합니다.

1. 응답 모델 확장

lib/shared/models/response_models.dart

이제 응답 모델은 요약 텍스트만 담는 구조가 아니라, 상태와 follow-up 종료 여부까지 담는 구조가 됩니다.

enum CommandResponseStatus {
  success,
  warning,
  error,
}

class CommandResponseModel {
  const CommandResponseModel({
    required this.transcript,
    required this.summary,
    this.followUp,
    this.isError = false,
    this.pendingAction,
    this.pendingTarget,
    this.status = CommandResponseStatus.success,
    this.completesFollowUp = false,
  });

  final String transcript;
  final String summary;
  final String? followUp;
  final bool isError;
  final String? pendingAction;
  final String? pendingTarget;
  final CommandResponseStatus status;
  final bool completesFollowUp;
}

Agent에서 내려주는 payload도 같은 정보를 받을 수 있도록 확장합니다.

3. HTTP 요청과 fallback 응답 정리

lib/features/session/application/session_controller.dart

실제 모드는 _requestAgentResponse(...)에서 HTTP로 Agent에 요청합니다. 여기서 중요한 건 실패도 단순 예외가 아니라 UI가 소비 가능한 payload로 바꿔서 돌려준다는 점입니다.

Future<AgentCommandPayload> _requestAgentResponse({
  required String command,
  required String source,
}) async {
  if (_agentApiBaseUrl.isEmpty) {
    return AgentCommandPayload(
      transcript: command,
      summary: 'Agent 서버 주소가 아직 설정되지 않아 명령을 처리할 수 없습니다.',
      followUp: 'API 서버 주소를 설정한 뒤 다시 시도할까요?',
      isError: true,
      status: 'error',
      completesFollowUp: true,
      rawText: 'Missing AGENT_API_BASE_URL',
    );
  }

  final uri = Uri.parse('$_agentApiBaseUrl/agent/respond');

  try {
    final response = await http
        .post(
          uri,
          headers: const {
            'Content-Type': 'application/json',
            'Accept': 'application/json',
          },
          body: jsonEncode(_buildRequestBody(
            command: command,
            source: source,
          )),
        )
        .timeout(const Duration(seconds: 15));

    if (response.statusCode < 200 || response.statusCode >= 300) {
      return AgentCommandPayload(
        transcript: command,
        summary: 'Agent 서버 요청에 실패했습니다. (${response.statusCode})',
        followUp: '서버 상태를 확인한 뒤 다시 시도할까요?',
        isError: true,
        status: 'error',
        completesFollowUp: true,
        rawText: response.body,
      );
    }

    return _parseAgentResponse(response.body);
  } catch (error) {
    return AgentCommandPayload(
      transcript: command,
      summary: 'Agent 서버에 연결하지 못했습니다.',
      followUp: '네트워크 또는 API 주소를 확인한 뒤 다시 시도할까요?',
      isError: true,
      status: 'error',
      completesFollowUp: true,
      rawText: error.toString(),
    );
  }
}

즉 실패해도 state와 UI는 일관된 구조로 값을 받게 됩니다.

4. 요청 body에 context 포함

lib/features/session/application/session_controller.dart

후속 질문 흐름을 유지하려면 이전 문맥을 같이 보내야 합니다. 그래서 HTTP body에 context를 포함합니다.

Map<String, dynamic> _buildRequestBody({
  required String command,
  required String source,
}) {
  return {
    'command': command,
    'source': source,
    'context': {
      'last_summary': state.lastSummary,
      'last_follow_up': state.lastFollowUp,
      'pending_action': state.pendingAction,
      'pending_target': state.pendingTarget,
    },
  };
}

이 구조 덕분에 응, 네, 재생해줘 같은 짧은 follow-up 응답도 의미를 가질 수 있습니다.

5. status 문자열을 enum으로 바꾸기

lib/features/session/application/session_controller.dart
CommandResponseStatus _resolveStatus(AgentCommandPayload payload) {
  final rawStatus = _cleanText(payload.status)?.toLowerCase();

  switch (rawStatus) {
    case 'warning':
      return CommandResponseStatus.warning;
    case 'error':
      return CommandResponseStatus.error;
    case 'success':
      return CommandResponseStatus.success;
  }

  if (payload.isError == true) {
    return CommandResponseStatus.error;
  }

  return CommandResponseStatus.success;
}

앱 내부에서는 문자열보다 enum이 더 안전하고, 토스트 분기에도 바로 쓰기 좋습니다.

6. Agent payload를 CommandResponseModel로 변환

lib/features/session/application/session_controller.dart
CommandResponseModel _mapAgentPayloadToCommandResponse(
  AgentCommandPayload payload, {
  required String fallbackTranscript,
  required String fallbackSummary,
}) {
  final transcript = _cleanText(payload.transcript) ?? fallbackTranscript;
  final summary = _cleanText(payload.summary) ??
      _extractSummaryFromRaw(payload.rawText) ??
      fallbackSummary;
  final followUp = _cleanText(payload.followUp);
  final pendingAction = _cleanText(payload.pendingAction);
  final pendingTarget = _cleanText(payload.pendingTarget);
  final status = _resolveStatus(payload);
  final completesFollowUp = payload.completesFollowUp ?? false;

  return CommandResponseModel(
    transcript: transcript,
    summary: summary,
    followUp: followUp,
    isError: status == CommandResponseStatus.error,
    pendingAction: pendingAction,
    pendingTarget: pendingTarget,
    status: status,
    completesFollowUp: completesFollowUp,
  );
}

summary가 비어 있으면 raw text 첫 줄을 fallback으로 쓰고, 그래도 없으면 각 메서드의 기본 요약 문구를 사용합니다.

6-1. 이번에 실제로 추가된 코드 포인트

lib/features/session/application/session_controller.dart
  • _mapAgentPayloadToCommandResponse()가 payload.isError만 보지 않고 _resolveStatus(payload)로 enum 상태를 계산합니다.
  • CommandResponseModel 생성 시 status와 completesFollowUp를 함께 채웁니다.
  • 즉 Day 18의 핵심은 “모델 필드 추가”뿐 아니라 “매핑 규칙 자체가 바뀌었다”는 점입니다.

return CommandResponseModel(

  followUp: followUp,

  pendingAction: pendingAction,

7. follow-up 완료 시 pending 상태 자동 정리

lib/features/session/application/session_controller.dart

이번 단계에서 가장 중요한 보강 중 하나입니다.

void _applyResponse(CommandResponseModel response) {
  final clearsPending = response.completesFollowUp;

  state = state.copyWith(
    isBusy: false,
    lastSummary: response.summary,
    lastFollowUp: response.followUp,
    pendingAction: clearsPending ? null : response.pendingAction,
    pendingTarget: clearsPending ? null : response.pendingTarget,
  );
}

이렇게 해야 follow-up이 끝난 뒤 이전 pendingAction이 남아서 다음 짧은 입력을 잘못 잡는 문제를 막을 수 있습니다.

8. HomePage 토스트 분기

lib/features/home/presentation/home_page.dart

Day 18부터는 결과 토스트도 응답 상태에 맞춰 분기합니다.

void _showCommandResultToast(CommandResponseModel response) {
  switch (response.status) {
    case CommandResponseStatus.success:
      showAppToast(
        context,
        response.followUp ?? '결과를 읽어드립니다.',
        title: '작업 완료',
        state: AppToastState.success,
      );
      break;
    case CommandResponseStatus.warning:
      showAppToast(
        context,
        response.summary,
        title: '확인 필요',
        state: AppToastState.warning,
      );
      break;
    case CommandResponseStatus.error:
      showAppToast(
        context,
        response.summary,
        title: '작업 실패',
        state: AppToastState.error,
      );
      break;
  }
}

즉 실제 모드 결과 영역과 토스트가 같은 상태 모델을 기준으로 움직이게 됩니다.

이 부분이 문서에서 빠지면 Day 18이 모델 레벨 변경으로만 보일 수 있는데, 실제로는 home_page.dart가 그 상태를 사용자 피드백으로 연결하는 마지막 지점입니다.

9. 최종적으로 달라지는 사용자 경험

  • 성공 응답은 더 자연스럽게 “작업 완료”로 보입니다.
  • 연결 실패는 중앙 결과와 토스트 모두에서 “작업 실패”로 일관되게 보입니다.
  • follow-up이 끝난 뒤 pending 상태가 자동 정리되어 짧은 응답 흐름이 더 안전해집니다.
  • HTTP Agent가 status와 completes_follow_up를 내려주면 더 풍부한 UX를 만들 수 있습니다.

Day 18 완료 기준

  • CommandResponseStatus enum이 추가되었다.
  • CommandResponseModel에 status와 completesFollowUp가 추가되었다.
  • AgentCommandPayload도 같은 정보를 받을 수 있다.
  • HTTP fallback 응답이 더 자연스러운 summary/followUp/status를 가진다.
  • follow-up 완료 시 pending 상태가 자동으로 정리된다.
  • 실제 모드 토스트가 success / warning / error로 분기된다.
정리: 18일차는 실제 모드에서 “응답이 온다” 수준을 넘어서, 응답의 의미와 종료 상태를 UI가 더 정확하게 표현하도록 만드는 단계입니다.
