## 1. 먼저 큰 그림부터 이해하기

현재 프로젝트에는 자동화 실행 방식이 사실상 두 종류 있다.

### 1. 기존 JSON plan 방식
이 방식은 먼저 바깥에서 아주 구체적인 실행 계획을 만든다.

예를 들면 이런 식이다.

- 1번: 네이버 지도 열기
- 2번: 길찾기 버튼 클릭
- 3번: 출발지 입력창 클릭
- 4번: `송내역` 입력
- 5번: Enter 누르기

즉, 무엇을 누르고 무엇을 입력할지가 이미 거의 결정된 상태에서 실행기가 그대로 따라간다.

이 방식의 장점은 step이 정확하면 실행도 단순하다는 점이다.  
하지만 단점은 step을 잘못 만들면 실행기가 똑같이 틀린 행동을 한다는 것이다.

### 2. 현재 pattern executor 방식
이 방식은 반대로 “작업 의미”만 먼저 정한다.

예를 들면 이런 식이다.

- 이 요청은 `keyword_search`다
- 또는 이 요청은 `paired_lookup`이다
- query는 `오늘 날씨`
- source는 `송내역`
- target은 `서울역`

그 다음 실제 웹페이지를 보고,

- 지금 검색창이 있는지
- 자동완성 후보가 떠 있는지
- 결과 영역이 보이는지
- 길찾기 패널이 열린 상태인지

를 엔진이 판단해서 실행한다.

즉 쉽게 말하면:

- 기존 방식: “무엇을 클릭할지 이미 정해져 있음”
- 현재 방식: “무슨 작업인지 알고, 현재 화면을 보고 클릭 대상을 고름”

이 차이가 가장 중요하다.

## 2. 네이버 검색 시나리오 쉬운 설명

예시 요청:

`네이버에서 오늘 날씨 검색해줘`

이 요청은 현재 엔진에서 보통 `keyword_search`로 해석된다.

### Step 1. 네이버 메인 페이지 열기
먼저 브라우저를 열고 네이버 메인으로 들어간다.

이때 화면에는 보통 다음 요소가 보인다.

- 검색창
- 검색 버튼
- 뉴스/쇼핑/메일 같은 메인 메뉴

여기서 엔진은 “현재 화면이 입력 가능한 상태인가”를 본다.  
검색창이 이미 보인다면 `input_ready`에 가까운 상태라고 볼 수 있다.

### Step 2. `open_primary_entry`
이 step은 “검색 모드로 들어가기 위한 대표 UI를 연다”는 뜻이다.

그런데 네이버 메인처럼 이미 검색창이 보이는 경우에는 굳이 따로 뭘 누를 필요가 없다.  
그래서 이 step은 실제로는 skip될 수 있다.

즉 이 step은 항상 클릭하는 step이 아니라,  
“필요하면 진입하고, 이미 준비되어 있으면 생략하는 step”이다.

### Step 3. `fill_slot(query)`
이제 검색창을 찾아 `오늘 날씨`를 입력한다.

이때 엔진은 검색창을 selector 이름으로 고정해서 찾는 것이 아니라,

- role이 textbox/searchbox/combobox인지
- placeholder나 텍스트에 검색 의미가 있는지
- 현재 페이지에서 가장 대표적인 입력창인지

를 점수화해서 가장 적절한 입력칸을 고른다.

### Step 4. `choose_suggestion(query)`
검색어를 입력하면 자동완성 후보가 뜰 수도 있다.

예를 들어:

- 오늘 날씨
- 오늘 날씨 서울
- 오늘 날씨 부산

같은 후보가 나타날 수 있다.

이 step은 그런 후보 중에서 현재 query와 가장 가까운 항목을 확정하는 역할이다.  
만약 적절한 후보를 못 찾으면 Enter를 누르는 fallback도 사용할 수 있다.

### Step 5. `submit_primary(search)`
보통은 여기서 검색 버튼을 누르거나 Enter로 검색을 확정한다.

그런데 네이버는 경우에 따라 자동완성 선택만으로 이미 검색 결과 페이지로 넘어갈 수 있다.  
이럴 때 submit을 또 하면 사용자가 보기에는 “왜 같은 행동을 두 번 하지?”처럼 느껴질 수 있다.

그래서 현재 엔진에는 이미 결과가 보이면 submit을 skip하는 로직이 들어 있다.

쉽게 말하면:

- 아직 결과 페이지가 아니면 submit 실행
- 이미 결과 페이지라면 submit 생략

이다.

### Step 6. `read_results`
마지막으로 검색 결과 영역을 읽는다.

여기서는 페이지 전체 HTML을 다 읽는 것이 아니라,

- main
- section
- article
- result region으로 보이는 영역

같은 부분에서 실제 사용자가 보게 되는 결과 텍스트를 요약하려고 한다.

### 사용자가 체감하는 화면 흐름
사용자 눈에는 대략 이렇게 보인다.

1. 네이버 열림
2. 검색창에 `오늘 날씨` 입력됨
3. 자동완성 후보가 잠깐 뜨거나 바로 결과 페이지로 넘어감
4. 결과 페이지가 보임
5. 결과 내용 요약이 반환됨

## 3. 네이버지도 길찾기 시나리오 쉬운 설명

예시 요청:

`네이버 지도로 송내역에서 서울역 가는 경로 알려줘`

이 요청은 현재 엔진에서 보통 `paired_lookup`으로 해석된다.  
`paired_lookup`은 “출발지/도착지처럼 2개의 슬롯을 채워서 결과를 찾는 작업”이다.

### Step 1. 네이버지도 진입
브라우저가 `map.naver.com`으로 이동한다.

이때 초기 화면은 길찾기 패널이 아닐 수도 있다.

- 메인 검색 화면일 수도 있고
- 지도 중심 홈 화면일 수도 있다

그래서 다음 step이 중요하다.

### Step 2. `open_primary_entry(route)`
이 step은 길찾기 패널을 여는 역할이다.

즉 그냥 검색창에 바로 입력하는 것이 아니라,
먼저 “길찾기”라는 작업 모드로 들어가야 한다.

현재 바인더는 이 step에서:

- 길찾기
- 경로
- route
- directions

같은 라벨을 가진 버튼/탭/패널 진입 요소를 찾으려고 한다.

만약 이미 출발지/도착지 입력칸이 모두 보이는 상태라면 이 step은 skip될 수도 있다.

### Step 3. `fill_slot(source)`
길찾기 패널이 열리면 출발지 입력칸을 찾고 `송내역`을 입력한다.

여기서 중요한 점은,
출발지와 도착지 입력칸이 placeholder 없이 거의 비슷하게 생긴 경우도 있다는 것이다.

그래서 현재 엔진은:

- `source`라는 의미의 별칭
- 위치 순서
- 관계 그룹

을 같이 보고 “이게 첫 번째 입력칸이 출발지일 가능성이 높다”고 판단한다.

### Step 4. `choose_suggestion(source)`
`송내역`을 입력한 뒤 뜨는 자동완성 후보 중에서 송내역을 확정한다.

이 step이 중요한 이유는 네이버지도는 단순히 텍스트만 입력했다고 끝나지 않고,  
실제 장소 후보를 확정해야 다음 단계가 안정적으로 이어지기 때문이다.

### Step 5. `fill_slot(target)`
이번에는 도착지 입력칸에 `서울역`을 입력한다.

### Step 6. `choose_suggestion(target)`
`서울역` 후보를 확정한다.

### Step 7. `submit_primary(route)`
출발지와 도착지가 모두 확정되면 실제 길찾기를 실행한다.

이 단계에서는:

- 길찾기 버튼 클릭
- 또는 Enter fallback

같은 방식이 사용될 수 있다.

### Step 8. `read_results`
마지막으로 경로 결과 영역을 읽는다.

현재는 이 부분이 아직 가장 거친 구간이다.  
검색이나 단순 결과 페이지보다, 네이버지도 경로 결과는 패널 구조가 복잡하기 때문이다.

그래서 현재 구조는:

- 결과 영역을 우선 읽으려고 시도
- 못 찾으면 body fallback을 사용

하는 수준까지 들어가 있다.

즉 “길찾기 자체는 된다”가 먼저이고,  
“결과를 얼마나 예쁘게 구조화해서 읽느냐”는 아직 더 보강할 수 있는 단계다.

### 사용자가 체감하는 화면 흐름
1. 네이버지도 열림
2. 길찾기 패널 열림
3. 출발지 `송내역` 입력 및 확정
4. 도착지 `서울역` 입력 및 확정
5. 길찾기 실행
6. 경로 결과 패널 표시
7. 결과 요약 반환

## 4. 유튜브 검색 시나리오 쉬운 설명

예시 요청:

`유튜브에서 고양이 영상 찾아줘`

이 요청은 역시 `keyword_search`로 해석된다.

### Step 1. 유튜브 진입
브라우저가 `youtube.com`으로 이동한다.

### Step 2. `open_primary_entry(search)`
유튜브는 기본적으로 상단 검색창이 이미 있는 경우가 많다.  
따라서 별도의 검색 모드 진입 없이 skip될 가능성이 높다.

### Step 3. `fill_slot(query)`
검색창에 `고양이 영상`을 입력한다.

이때 유튜브는 비슷한 버튼이 많아서,

- 검색 버튼
- 마이크 버튼
- 검색어 지우기 버튼

을 헷갈릴 수 있다.

그래서 현재 엔진에서는 입력창과 검색 버튼을 고를 때 “검색어 삭제”, “음성” 같은 부정 라벨에 패널티를 주고 있다.

### Step 4. `choose_suggestion(query)`
자동완성 추천 목록이 뜨면 실제 option 항목을 우선 선택한다.

이걸 보강한 이유는, 예전에는 listbox 전체를 눌러버리거나 애매한 영역을 눌러서 동작이 흔들릴 수 있었기 때문이다.

### Step 5. `submit_primary(search)`
이제 진짜 검색 버튼을 클릭한다.

현재 엔진은 이 단계에서:

- clear 버튼
- voice 버튼
- selected tab

같은 잘못된 대상을 피하도록 점수를 깎고 있다.

### Step 6. `read_results`
유튜브 결과 페이지로 이동한 뒤 결과 영역을 읽는다.

아직 “첫 번째 영상 카드만 정확히 구조화해서 읽는 수준”은 아니고,  
현재는 결과 영역 요약에 더 가깝다.

즉 지금 단계는:

- 검색은 비교적 안정적으로 실행됨
- 결과 카드 단위 정밀 추출은 향후 보강 가능

으로 이해하면 된다.

### 사용자가 체감하는 화면 흐름
1. 유튜브 열림
2. 검색창에 `고양이 영상` 입력
3. 자동완성 후보 확정 또는 Enter
4. 검색 버튼 클릭
5. 결과 페이지 이동
6. 결과 요약 반환

## 5. “지금 하드코딩이 아예 없는 거야?”에 대한 쉬운 설명

이 질문은 아주 중요하다.  
답은 **“사이트별 selector 하드코딩은 새 경로에는 거의 없지만, 완전히 무하드코딩은 아니다”** 이다.

### 5.1 없는 것
현재 pattern executor 경로에는 이런 것이 없다.

- `if naver map then click #some-selector`
- `if youtube then fill input[name=search_query]`
- `NaverMapRouteScript` 같은 사이트 전용 매크로 클래스
- LLM이 Playwright 코드를 직접 작성하는 구조

즉 “네이버면 무조건 이 CSS를 써라” 같은 식의 새 엔진 전용 코드는 없다.

### 5.2 있는 것
하지만 범용 엔진을 만들려면 최소한의 정책값은 들어가야 한다.

현재 들어가 있는 것은 이런 것들이다.

#### 1. 작업 종류는 고정
현재 task ontology는 예를 들면 아래처럼 정해져 있다.

- `keyword_search`
- `paired_lookup`
- `form_fill`

즉 작업 의미 체계 자체는 코드에 정의되어 있다.

#### 2. action 순서 템플릿은 고정
예를 들어 `keyword_search`는 대체로:

- open_primary_entry
- fill_slot
- choose_suggestion
- submit_primary
- read_results

순서로 간다.

이건 사이트별 하드코딩은 아니지만,  
“검색형 작업은 보통 이런 흐름을 가진다”는 의미의 템플릿 하드코딩이다.

#### 3. 슬롯 별칭과 라벨 사전이 고정
예를 들어:

- `source` = 출발, 출발지, origin, departure
- `target` = 도착, 도착지, destination, arrival
- `route` = 길찾기, 경로, direction
- `search` = 검색, search, find

이런 단어 사전이 바인더 안에 들어 있다.

이건 사이트별 하드코딩이 아니라,  
UI 의미를 이해하기 위한 패턴 사전이다.

#### 4. 임계값이 고정
예:

- confidence가 0.8 이상이면 자동 실행
- 0.5 이상이면 조심스럽게 진행
- 그보다 낮으면 복구 또는 실패 처리

이런 기준도 정책값으로 고정돼 있다.

#### 5. 결과 영역/대화상자 탐지용 범용 selector는 있음
예를 들어:

- `dialog,[role="dialog"]`
- `main,[role="main"],section,article,[role="region"],[role="tabpanel"]`

같은 범용 selector로 region을 수집한다.

이건 특정 사이트를 위한 selector가 아니라,  
“어떤 게 결과 영역일 가능성이 높은가”를 판단하기 위한 공통 탐지 규칙이다.

### 5.3 한 줄 정리
정확히 말하면 현재 엔진은:

- 사이트별 하드코딩 엔진은 아님
- 하지만 정책/라벨/별칭/임계값이 들어간 패턴 기반 엔진임

즉 “완전한 제로 하드코딩”은 아니고,  
“사이트 전용 매크로가 없는 제한적 패턴 하드코딩”이라고 이해하면 가장 정확하다.

## 6. connected mode와 어떻게 연결되는가

기존 connected mode는 이미 실제 API를 호출하고 있었다.

흐름은 대략 이렇다.

1. Flutter 앱이 API에 명령을 보냄
2. API가 응답을 돌려줌
3. 그 응답이 legacy JSON plan이면
4. 앱이 Python 실행기를 띄워 자동화 실행

즉 “실제 API 연동” 자체는 예전부터 있었다.

### 이번에 바뀐 점
이제 connected mode가 legacy JSON plan뿐 아니라  
**pattern task 형태의 응답도 받아서 실행**할 수 있게 되었다.

즉 API가 앞으로 꼭 저수준 `steps[]`를 반환하지 않아도 된다.

대신 이런 식으로 반환하면 된다.

- site
- user_request
- intent
- host_bias
- metadata

그럼 Flutter가 이걸 감지해서 새 `PatternAutomationRunnerService`로 넘기고,  
Python 쪽 `pattern_agent_scenario.py`가 실행되면서 새 패턴 엔진으로 동작한다.

### 왜 이게 중요한가
이제 API 서버 입장에서는 더 이상:

- click
- fill
- press
- wait_for

같은 저수준 step을 굳이 직접 만들 필요가 줄어든다.

대신:

- 이건 검색이다
- 이건 길찾기다
- query는 이거다
- source/target은 이거다

정도만 정확히 반환하면, 실행은 로컬 패턴 엔진이 맡을 수 있다.

이게 바로 비용과 안정성을 동시에 잡으려는 방향이다.

## 7. 기존 API 응답과 현재 추천 응답의 차이

### 7.1 기존 방식
기존 API는 보통 이런 느낌의 응답을 돌려준다.

```json
{
  "task_id": "naver_map_route",
  "goal": "송내역에서 서울역 가는 경로",
  "site": "https://map.naver.com",
  "steps": [
    { "step": 1, "type": "navigate", "args": { "url": "https://map.naver.com" } },
    { "step": 2, "type": "click", "target": { "description": "길찾기 버튼" } }
  ]
}
```

이 방식은 API가 실행 step을 아주 많이 책임진다.

즉 서버가 실수하면 실행도 그대로 흔들린다.

### 7.2 현재 추천 방식
현재 추천하는 방식은 이런 느낌이다.

```json
{
  "site": "https://map.naver.com",
  "user_request": "네이버 지도로 송내역에서 서울역 가는 경로 알려줘",
  "intent": {
    "task_type": "paired_lookup",
    "slots": {
      "source": "송내역",
      "target": "서울역"
    },
    "risk_level": "low",
    "confidence": 0.94
  },
  "host_bias": {
    "host": "map.naver.com",
    "prefer_panel_ui": 1.2,
    "autocomplete_confirm_weight": 1.15,
    "primary_action_label_preferences": ["route"]
  }
}
```

이 응답은 “의미”만 강하게 담고 있다.

즉 서버는:

- 무엇을 하고 싶은지
- 어떤 값이 중요한지
- 약간의 힌트가 있는지

만 말해주고,

실제로 어느 버튼을 누르고 어떤 입력창을 찾을지는 클라이언트 쪽 패턴 엔진이 결정한다.

### 왜 이게 더 낫나
장기적으로는 이 방식이 더 낫다.

이유는:

- API가 저수준 step을 매번 만들지 않아도 되고
- 같은 요청에 대해 응답이 덜 흔들리고
- 로컬 엔진 개선으로 여러 사이트에 공통 효과를 낼 수 있기 때문이다

---

## 8. 현재 기준으로 “반환해야 할 값”은 무엇인가

여기서 말하는 “반환해야 할 값”은 두 방향이 있다.

### 8.1 API가 앱에 반환해야 할 값
현재 pattern executor를 쓰려면 API는 최소한 아래를 돌려주는 것이 좋다.

- `site`
- `user_request`
- `intent.task_type`
- `intent.slots`
- `intent.risk_level`
- `intent.confidence`

선택적으로:

- `host_bias`
- `metadata`

도 줄 수 있다.

### 8.2 실행기가 실행 후 돌려주는 값
현재 pattern executor가 성공하면 대략 아래 값을 반환한다.

- `status`
- `scenario`
- `goal`
- `summary`
- `url`
- `title`
- `steps`
- `observations`
- `intent`
- `snapshot`
- `engine`

즉 기존보다 더 풍부하다.

특히 중요한 것은:

- 실제로 어떤 intent로 해석했는지
- step마다 어떤 상태였는지
- 어떤 후보를 골랐는지

같은 디버깅 정보가 더 많이 담긴다는 점이다.

### 8.3 앱에서 최종 사용자에게 보여줄 때 중요한 값
실제로 사용자에게 당장 중요한 것은 아래다.

- 성공/실패 여부
- 한 줄 요약 summary
- 필요시 follow-up

하지만 개발과 디버깅 관점에서는 아래가 중요하다.

- intent
- steps
- observations
- snapshot.state

즉 사용자용 응답과 개발용 응답은 목적이 다르다.  
한 API 응답 안에 둘 다 담되, UI에서 노출 범위만 나누는 것이 좋다.

---

## 9. LLM 프롬프트는 왜 바뀌어야 하는가

기존 방식에서는 LLM이 사실상 step 생성기 역할을 하기도 했다.

예를 들면:

- 길찾기 버튼 클릭
- 출발지 입력
- Enter
- 도착지 입력

같은 저수준 실행을 많이 책임졌다.

하지만 pattern executor 구조에서는 이 역할을 줄여야 한다.

### LLM이 해야 하는 일
LLM은 아래 정도만 맡는 것이 좋다.

- 이 요청이 `keyword_search`인지 `paired_lookup`인지 판단
- query/source/target 같은 slot 추출
- risk level 설정
- 필요할 때만 가벼운 host bias 추가

### LLM이 하면 안 되는 일
아래는 이제 하지 않는 것이 좋다.

- CSS selector 생성
- Playwright 코드 생성
- click/fill/press step 나열
- 사이트별 매크로 시퀀스 생성

### 왜 그렇게 해야 하나
이유는 단순하다.

- selector를 LLM이 만들면 흔들리기 쉽다
- 코드 생성은 비용이 커진다
- 같은 요청에도 step이 달라질 수 있다

반대로 task/slot만 뽑으면:

- 응답이 짧아진다
- 비용이 줄어든다
- 실행은 결정론적 엔진이 맡으므로 안정성이 올라간다

## 10. 현재 구조를 한 문장으로 요약하면

현재 구조는 다음 한 문장으로 요약할 수 있다.

**“LLM은 사용자의 의도를 `작업 의미`로 정리하고, 실제 웹페이지에서 무엇을 클릭하고 입력할지는 로컬 패턴 엔진이 현재 화면을 보고 결정한다.”**

이 말이 곧 이번 개선안의 핵심이다.

## 11. 실무 관점에서 지금 상태를 어떻게 봐야 하나

현재 상태를 너무 낙관적으로 보면 안 되고, 그렇다고 과소평가할 필요도 없다.

### 이미 확보한 것
- 검색형 작업은 여러 사이트에 공통으로 재사용 가능한 기반이 생김
- 네이버 검색, 네이버지도 길찾기, 유튜브 검색처럼 성격이 다른 사례를 하나의 엔진으로 다룰 수 있게 됨
- connected mode에서도 pattern task 응답을 받아 실제 API 연동 테스트를 할 수 있게 됨

### 아직 더 보강할 것
- 결과 요약 품질, 특히 복잡한 결과 패널 추출
- first result click 같은 후속 행동
- pattern ontology 확장
- 실제 LLM 연동 시 응답 안정성 검증

즉 지금 단계는:

- “방향 검증”은 꽤 됨
- “완성형 제품 수준”까지는 아직 보강 여지 있음

이라고 보는 것이 맞다.
