1. Day 0에서 이 프로젝트는 어디까지 와 있었나

Day 0 시점의 이 프로젝트는 “완성된 앱”이라기보다 실제 모드 UI 초안 + 상태 흐름 초안 + 설정 저장 초안까지 만들어진 상태였습니다.

  • 실제 모드로 들어가는 기본 홈 화면
  • 음성 듣기 / 화면 읽기 / 텍스트 명령 흐름

  • 로컬 설정 저장과 설정 모달

아직 없던 것

  • 데모 모드 화면
  • 완성형 모드 선택 화면

핵심 해석: Day 0 프로젝트는 “기능 개념은 잡혀 있고, 디자인 시스템은 아직 약한 상태”였습니다.

3. pubspec.yaml 해설

Flutter 프로젝트에서 pubspec.yaml은 “이 앱이 어떤 이름을 가지고, 어떤 패키지를 쓰고, 어떤 폰트를 등록하는지”를 정의하는 중심 파일입니다. 

name: navi_voice_navigator
description: Voice Navigator accessibility desktop assistant
publish_to: none
version: 0.1.0+1

environment:
  sdk: ">=3.4.0 <4.0.0"

dependencies:
  flutter:
    sdk: flutter
  cupertino_icons: ^1.0.8
  flutter_riverpod: ^2.5.1

dev_dependencies:
  flutter_test:
    sdk: flutter
  flutter_lints: ^4.0.0
항목 뜻
name 프로젝트 이름입니다. 패키지 식별에도 영향을 줍니다.
description 앱 설명입니다. 문서적 의미가 큽니다.
publish_to: none pub.dev에 공개 배포하지 않는 로컬 앱이라는 뜻입니다.
environment 이 프로젝트가 어느 Dart SDK 범위에서 동작하는지 지정합니다.
dependencies 앱 실행에 필요한 패키지 목록입니다.
dev_dependencies 테스트나 린트처럼 개발 단계에서만 필요한 패키지입니다.
fonts Pretendard 폰트를 앱 전체에서 쓸 수 있게 등록합니다.

왜 material.dart가 필요한가

대부분의 Dart 파일 맨 위에 있는 import 'package:flutter/material.dart';는 Flutter의 Material UI 도구 상자를 가져오는 구문입니다.

  • MaterialApp: 앱 전체 틀
  • Scaffold: 화면의 기본 뼈대
  • Text, Row, Column: 가장 기본적인 UI 조립 부품
  • ElevatedButton, OutlinedButton: 버튼
  • TextField, Icon, Container: 입력창, 아이콘, 박스
쉽게 말하면: material.dart는 Flutter 화면을 만드는 기본 공구 세트입니다.

왜 flutter_riverpod가 필요한가

이 프로젝트는 단순 정적 화면이 아니라, 음성 듣기 상태, 처리 상태, 설정 값처럼 계속 바뀌는 데이터를 다뤄야 합니다. 그래서 상태 관리가 필요합니다.

Riverpod은 상태를 화면 밖에서 관리하고, 필요한 화면만 그 상태를 읽게 해주는 도구입니다.

  • ProviderScope: Riverpod이 전체 앱에서 동작하도록 감싸는 루트
  • StateNotifier: 상태를 변경하는 컨트롤러 클래스
  • StateNotifierProvider: 컨트롤러와 상태를 화면에 연결하는 통로
  • ref.watch(...): 상태를 구독해서 값이 바뀌면 화면 다시 그림
  • ref.read(...): 한 번 읽거나 액션 메서드를 호출할 때 사용
쉽게 말하면: Riverpod은 “앱 상태를 중앙에서 관리하고 화면이 그 값을 구독하게 하는 시스템”입니다.

4. 앱 시작 흐름

main.dart

void main() {
  WidgetsFlutterBinding.ensureInitialized();
  runApp(const ProviderScope(child: VoiceNavigatorApp()));
}

이 파일은 앱의 진짜 시작점입니다.

  1. WidgetsFlutterBinding.ensureInitialized()로 Flutter 엔진 준비
  2. ProviderScope로 Riverpod 범위 생성
  3. VoiceNavigatorApp를 실행

app_shell.dart

이 파일은 “앱 전체 컨테이너”입니다. Day 0 기준으로는 아직 테마는 없지만, 설정을 먼저 로드한 다음 메인 화면으로 진입시키는 역할을 하고 있습니다. 

class VoiceNavigatorApp extends ConsumerStatefulWidget { ... }
  • ConsumerStatefulWidget인 이유: Riverpod 상태도 읽고 initState()도 써야 하기 때문
  • _loaded: 설정 로드가 끝났는지 표시하는 플래그
  • settingsControllerProvider.notifier.load(): 저장된 설정 불러오기
  • 로딩 중에는 CircularProgressIndicator 표시
Day 0의 app_shell.dart는 이미 “설정을 먼저 읽고 앱을 시작한다”는 좋은 구조를 갖고 있었습니다.

6. 실제 모드 홈 화면 분석

Day 0 기준 핵심 화면은 [홈 화면]이었습니다. 이 화면은 하나의 거대한 위젯이라기보다, 여러 작은 위젯을 조립해 만든 구조입니다.

home_page.dart의 책임

  • 상태 읽기: 듣기 상태, 세션 상태, 설정 상태
  • 이벤트 처리: 음성 듣기, 화면 읽기, 텍스트 명령 제출
  • 레이아웃 조립: 타이틀바, 상태카드, 액션패널, 준비영역, 설정모달

왜 ConsumerStatefulWidget인가

이 화면은 Riverpod 상태를 읽어야 하고, 동시에 _showSettings 같은 로컬 UI 상태도 직접 관리해야 합니다. 그래서 ConsumerWidget보다 ConsumerStatefulWidget이 적합했습니다.

핸들러 메서드의 역할

메서드 역할
_handleListen() 녹음 시작 또는 녹음 종료 후 처리로 전환
_handleScreenRead() 현재 화면 읽기 요청
_handleSubmitText() 텍스트 명령을 세션 컨트롤러로 전달
_micLabel() 상태 enum을 사람이 읽는 한글 문자열로 변환
중요한 구조 포인트: 화면은 직접 모든 비즈니스 로직을 처리하지 않고, Riverpod 컨트롤러를 호출해서 상태만 바꾸도록 되어 있습니다.

7. 홈 화면을 이루는 작은 위젯들

title_bar.dart

상단 타이틀 영역입니다. 앱 이름, 아이콘, 비활성 윈도우 제어 버튼 모양을 보여줍니다. 실제 기능보다 “데스크톱 앱처럼 보이게 만드는 시각 요소” 역할이 큽니다.

status_card.dart

마이크 상태, 현재 모드처럼 “작은 정보 카드”를 재사용 가능하게 만든 위젯입니다. 라벨, 값, 아이콘을 받아서 공통 UI를 출력합니다.

action_panel.dart

왼쪽 버튼 모음 영역입니다. 음성 듣기, 화면 읽기, 설정 열기 버튼이 있고, 화면에서 받은 콜백을 실행합니다. 여기서는 로직보다 “사용자 액션 전달”이 핵심입니다.

ready_state.dart

중앙 메인 영역입니다. 앱 상태 설명, 처리 중 문구, 결과 요약 카드, 텍스트 명령 입력 컴포넌트를 함께 보여줍니다.

text_command_composer.dart

텍스트 명령 입력창과 제출 버튼입니다. TextEditingController를 이용해 입력값을 관리하고, 제출 시 부모가 넘긴 콜백을 호출합니다.

8. 상태 관리 파일 분석

listening_controller.dart

이 파일은 “마이크 상태”만 관리합니다.

  • idle: 대기 상태
  • listening: 음성을 듣는 중
  • processing: 명령 처리 중

상태가 단순해서 ListeningState도 매우 작습니다. 이건 좋은 신호입니다. 상태를 잘게 나누면 이해가 쉬워집니다.

session_controller.dart

이 파일은 실제 모드의 “결과 상태”를 관리합니다.

  • lastSummary: 마지막 결과 요약
  • lastFollowUp: 후속 질문
  • isBusy: 처리 중 여부
  • isRecording: 녹음 중 여부

Day 0 기준 이 컨트롤러는 진짜 백엔드 호출 대신 Future.delayed로 가짜 응답을 만들고 있었습니다. 이것은 개발 초기에 매우 유용한 방식입니다.

의미: UI를 먼저 만들고, 나중에 실제 서버 연동으로 바꾸기 쉬운 구조입니다.

response_models.dart

세션 결과를 담는 작은 데이터 모델입니다. 문자열 3개만 있지만, 이런 식으로 별도 모델을 두면 나중에 응답 구조가 커져도 정리가 쉽습니다.

9. 설정 시스템 분석

settings_models.dart

Day 0 기준 설정 모델은 이미 꽤 괜찮은 구조로 나뉘어 있었습니다.

  • AppSettings: 전체 설정 묶음
  • GeneralSettings: 일반 설정
  • ShortcutSettings: 단축키
  • SecuritySettings: 보안 설정
  • DisplaySettings: 화면 표시 설정

여기서 가장 중요한 패턴은 copyWith()와 toJson()/fromJson()입니다.

  • copyWith(): 상태 일부만 바꾸기 좋음
  • toJson(): 파일 저장할 때 필요
  • fromJson(): 파일에서 다시 읽을 때 필요

settings_controller.dart

설정 상태를 변경하는 Riverpod 컨트롤러입니다. 각 토글과 입력값을 바꿀 때 직접 필드를 만지는 대신, 메서드를 통해 상태를 바꾸도록 설계되어 있습니다.

local_settings_store.dart

설정을 로컬 파일 runtime/settings.json에 저장하고 다시 읽는 서비스입니다.

  • 파일이 없으면 기본값 사용
  • 파일이 깨졌어도 기본값으로 복구
  • 저장 시 폴더가 없으면 자동 생성

이건 초기에 매우 좋은 선택입니다. DB 없이도 설정 persistence를 쉽게 구현할 수 있기 때문입니다.

settings_modal.dart

설정 UI를 보여주는 모달입니다.

  • ConsumerWidget이므로 설정 상태를 바로 읽을 수 있음
  • SwitchListTile로 토글 설정 제어
  • TextField로 단축키 텍스트 입력
  • 저장 버튼 클릭 시 파일 저장 후 모달 닫기

다만 Day 0 기준 설정 화면은 기능은 있지만, 시각적으로는 아직 목표 프로젝트 수준까지 정리되지는 않은 상태였습니다.

10. Day 0에서 꼭 이해해야 하는 Flutter 개념

개념 왜 이 프로젝트에서 중요한가
StatelessWidget 단순 UI 조각을 만들 때 사용합니다. 예: StatusCard, AppTitleBar
StatefulWidget 내부 상태가 변하는 UI에 필요합니다. 예: 텍스트 입력창
ConsumerWidget / ConsumerStatefulWidget Riverpod 상태를 읽어야 할 때 필요합니다.
Scaffold 페이지의 기본 뼈대를 만들어 줍니다.
Column, Row, Expanded 데스크톱형 레이아웃을 만드는 기본 조합입니다.
Navigator.push 런처 화면에서 실제 모드 화면으로 이동할 때 사용합니다.
TextEditingController 입력창의 현재 텍스트를 읽고 지우기 위해 필요합니다.
Future / async / await 설정 로드, 가짜 응답 처리처럼 시간이 걸리는 작업을 다루기 위해 필요합니다.

11. Day 0 기준 프로젝트의 강점과 약점

강점

  • 기능 단위로 파일이 나뉘어 있음
  • Riverpod 상태 분리가 이미 시작됨
  • 설정 저장 구조가 있음
  • 홈 화면이 위젯 단위로 잘 분리됨
  • 실제 모드 흐름을 가짜 데이터로 먼저 구현해 둠

약점

  • 공통 테마 시스템이 없음
  • 데모 모드가 아직 없음
  • 모드 선택 화면이 임시 형태
  • 알림/팝업 UI가 없음
  • 설정 화면의 구조와 시각 완성도가 아직 낮음

13. Day 0에서 Day 1으로 넘어가는 이유

Day 0까지는 구조와 기능 초안이 있었습니다. 하지만 목표 프로젝트처럼 보이기 위해서는 이제 디자인 시스템과 공통 테마가 필요합니다.

  • 왜냐하면 다크 테마, 큰 글씨, 고대비 같은 옵션이 생기기 시작했기 때문이고
  • 앞으로 모드 선택 화면, 실제 모드, 데모 모드가 모두 같은 스타일을 써야 하기 때문입니다.

그래서 Day 1의 핵심은 “새 화면 많이 만들기”가 아니라, “공통 기반을 만드는 것”이었습니다.
