Navi Voice Navigator

프로젝트 신규 생성부터 6일차까지, 실제로 따라칠 수 있는 형태로 정리한 통합 튜토리얼입니다.

Sections
Project Setup

신규 프로젝트 생성과 기본 세팅

여기서는 Flutter 프로젝트를 새로 만든 뒤, 현재 이 저장소의 기본 구조까지 만드는 과정을 설명합니다.

목표

  • Flutter 프로젝트를 새로 만든다.
  • flutter_riverpod와 Pretendard 폰트를 추가한다.
  • main.dart를 Riverpod 기반 시작점으로 바꾼다.
  • 앱 폴더 구조를 미리 잡아 이후 날짜별 작업이 쉬운 상태를 만든다.

작업 순서

프로젝트를 생성한다

flutter create navi_voice_navigator

폰트 파일을 assets/fonts에 넣는다

Pretendard Regular/Medium/SemiBold/Bold OTF 파일을 assets/fonts/에 복사합니다.

pubspec.yaml을 수정한다

Riverpod 의존성과 폰트 등록을 먼저 끝내야 이후 테마 작업이 자연스럽습니다.

main.dart를 교체한다

앱 시작점을 ProviderScope 기반으로 바꿉니다.

파일 1: pubspec.yaml

pubspec.yaml

이 파일은 프로젝트 이름, Dart SDK 버전, 패키지 의존성, 폰트 등록 같은 앱 기본 메타데이터를 관리합니다.

먼저 기존 pubspec.yaml을 열고, dependencies와 flutter/fonts 영역을 아래처럼 맞춥니다. 

name: navi_voice_navigator
description: Voice Navigator accessibility desktop assistant
publish_to: none
version: 0.1.0+1

environment:
  sdk: ">=3.4.0 <4.0.0"

dependencies:
  flutter:
    sdk: flutter
  cupertino_icons: ^1.0.8
  flutter_riverpod: ^2.5.1

dev_dependencies:
  flutter_test:
    sdk: flutter
  flutter_lints: ^4.0.0

flutter:
  uses-material-design: true

  fonts:
    - family: Pretendard
      fonts:
        - asset: assets/fonts/Pretendard-Regular.otf
          weight: 400
        - asset: assets/fonts/Pretendard-Medium.otf
          weight: 500
        - asset: assets/fonts/Pretendard-SemiBold.otf
          weight: 600
        - asset: assets/fonts/Pretendard-Bold.otf
          weight: 700

수정 후에는 반드시 flutter pub get을 실행합니다.

파일 2: main.dart

lib/main.dart

이 파일은 앱의 가장 바깥 시작점입니다. 여기서 ProviderScope로 감싸면 Riverpod 상태를 프로젝트 전체에서 사용할 수 있습니다.

기존 내용을 지우고 아래 코드로 교체합니다.

import 'package:flutter/material.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';

import 'app/app_shell.dart';

void main() {
  WidgetsFlutterBinding.ensureInitialized();
  runApp(const ProviderScope(child: VoiceNavigatorApp()));
}

파일 4: listening_controller.dart

여기서는 마이크 상태만 따로 관리합니다. Day 0에서는 idle, listening, processing 3단계만 있으면 충분합니다. 

import 'package:flutter_riverpod/flutter_riverpod.dart';

enum ListeningStatus { idle, listening, processing }

class ListeningState {
  const ListeningState(this.status);

  final ListeningStatus status;
}

class ListeningController extends StateNotifier<ListeningState> {
  ListeningController() : super(const ListeningState(ListeningStatus.idle));

  void startListening() {
    state = const ListeningState(ListeningStatus.listening);
  }

  void setProcessing() {
    state = const ListeningState(ListeningStatus.processing);
  }

  void reset() {
    state = const ListeningState(ListeningStatus.idle);
  }
}

final listeningControllerProvider =
    StateNotifierProvider<ListeningController, ListeningState>((ref) {
  return ListeningController();
});

파일 5: session_controller.dart

lib/features/session/application/session_controller.dart

Day 0에서는 실제 API 연결 대신 더미 응답으로도 충분합니다. 중요한 건 “상태 구조”를 먼저 만드는 것입니다. 

import 'package:flutter_riverpod/flutter_riverpod.dart';

import '../../../shared/models/response_models.dart';

class SessionUiState {
  const SessionUiState({
    this.lastSummary,
    this.lastFollowUp,
    this.isBusy = false,
    this.isRecording = false,
  });

  final String? lastSummary;
  final String? lastFollowUp;
  final bool isBusy;
  final bool isRecording;

  SessionUiState copyWith({
    String? lastSummary,
    String? lastFollowUp,
    bool? isBusy,
    bool? isRecording,
  }) {
    return SessionUiState(
      lastSummary: lastSummary ?? this.lastSummary,
      lastFollowUp: lastFollowUp ?? this.lastFollowUp,
      isBusy: isBusy ?? this.isBusy,
      isRecording: isRecording ?? this.isRecording,
    );
  }
}

class SessionController extends StateNotifier<SessionUiState> {
  SessionController() : super(const SessionUiState());

  Future<void> startListening() async {
    state = state.copyWith(isRecording: true);
  }

  Future<CommandResponseModel> stopListeningAndProcess() async {
    state = state.copyWith(isRecording: false, isBusy: true);
    await Future<void>.delayed(const Duration(milliseconds: 900));

    const response = CommandResponseModel(
      transcript: '유튜브에서 고양이 영상 찾아줘',
      summary: '유튜브 검색 결과를 준비했습니다.',
      followUp: '첫 번째 결과를 재생할까요?',
    );

    state = state.copyWith(
      isBusy: false,
      lastSummary: response.summary,
      lastFollowUp: response.followUp,
    );

    return response;
  }
}

final sessionControllerProvider =
    StateNotifierProvider<SessionController, SessionUiState>((ref) {
  return SessionController();
});

Day 1

Day 1의 핵심은 “앞으로 만들 모든 화면이 공통으로 기대는 기반” 을 만드는 것입니다.

파일 2: home_page.dart

lib/features/home/presentation/home_page.dart

먼저 import를 추가합니다. 

import '../../notifications/presentation/app_toast.dart';

그 다음 _handleListen() 안에 아래 3군데를 넣습니다.

showAppToast(
  context,
  '음성을 듣고 있습니다. 말씀해주세요.',
  title: '음성 수신 중',
  state: AppToastState.listening,
);
showAppToast(
  context,
  '작업을 처리하고 있습니다.',
  title: '작업 처리 중',
  state: AppToastState.processing,
);
showAppToast(
  context,
  '결과를 읽어드립니다.',
  title: '작업 완료',
  state: AppToastState.success,
);

_handleScreenRead()와 _handleSubmitText()에도 같은 패턴으로 처리중/완료 토스트를 연결합니다.

파일 3: demo_home_page.dart

lib/demo/demo_home_page.dart

실제 모드와 같은 토스트 경험을 만들기 위해 데모 모드에도 동일한 함수를 연결합니다. 

import '../features/notifications/presentation/app_toast.dart';

그리고 _simulateListen(), _simulateScreenRead(), _handleTextCommand() 안에 각각 듣는 중 / 처리 중 / 완료 토스트를 넣습니다.

Day 6 완료 기준
  • 실제 모드에서 듣기 시작, 처리 중, 완료 토스트가 뜬다.
  • 데모 모드도 같은 토스트 흐름을 사용한다.
  • 토스트는 일정 시간 후 자동으로 사라진다.
  • 앱이 훨씬 반응적이고 살아 있는 느낌을 준다.
