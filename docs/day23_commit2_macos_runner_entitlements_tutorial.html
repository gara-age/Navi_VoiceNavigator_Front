Commit 2

macOS Runner Entitlements

이 문서는 chore: update macos runner entitlements 커밋을 재구현하기 위한 튜토리얼입니다. 수정 범위는 작지만, 왜 이 권한이 필요한지와 최종 plist 내용을 정확히 이해하고 반영하는 것이 중요합니다.

주의: Entitlements는 기능 추가가 아니라 Runner 권한 설정입니다. 값 하나가 잘못되어도 macOS 런타임 동작이 달라지므로, 문서의 최종 plist를 그대로 맞추는 것이 안전합니다.

1. 이번 커밋에서 건드리는 파일

  • macos/Runner/DebugProfile.entitlements
  • macos/Runner/Release.entitlements

즉 이번 커밋은 macOS Runner 권한 설정만 다룹니다. Flutter Dart 코드나 UI 코드는 변경하지 않습니다.

중요한 점은 이 커밋에서 project.pbxproj를 함께 수정하지 않는다는 것입니다. 현재 Runner 타깃이 이미 이 entitlement 파일들을 참조하고 있다는 전제가 있는 정리 커밋입니다.

2. entitlements를 왜 따로 두는가

macOS 앱은 Debug와 Release에서 필요한 권한 구성이 다를 수 있습니다. 특히 Flutter debug 런타임은 JIT와 개발용 네트워크 동작을 사용할 수 있기 때문에, Debug 프로파일에서 더 많은 권한이 필요할 수 있습니다.

  • DebugProfile.entitlements: 개발 실행과 디버그 런타임을 위한 권한
  • Release.entitlements: 배포 빌드에서 필요한 최소 권한

3. DebugProfile.entitlements 최종 코드

macos/Runner/DebugProfile.entitlements
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>com.apple.security.cs.allow-jit</key>
  <true/>
  <key>com.apple.security.network.client</key>
  <true/>
  <key>com.apple.security.network.server</key>
  <true/>
</dict>
</plist>

여기서 각 key의 의미는 다음과 같습니다.

  • com.apple.security.cs.allow-jit: Flutter debug 런타임의 JIT 허용
  • com.apple.security.network.client: 외부 HTTP/WS 호출 허용
  • com.apple.security.network.server: 디버그 환경에서 로컬 서버/수신 동작 허용

4. Release.entitlements 최종 코드

macos/Runner/Release.entitlements
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>com.apple.security.network.client</key>
  <true/>
</dict>
</plist>

Release에서는 네트워크 client 권한만 남깁니다. 이 프로젝트의 Release 빌드는 외부/로컬 서버에 요청을 보낼 수 있으면 되고, Debug처럼 JIT나 서버 권한이 필요하지 않기 때문입니다.

5. 왜 Debug와 Release가 다른가

  • Debug는 개발 중이라 Flutter 엔진과 디버거 요구사항이 더 큽니다.
  • Release는 실제 실행에 필요한 최소 권한만 두는 편이 안정적입니다.
  • 특히 allow-jit는 Debug에서는 자연스럽지만 Release에서는 불필요할 수 있습니다.
실무 팁: macOS에서 네트워크 호출, 로컬 Python 보조 프로세스 연동, 디버그 실행이 섞이면 entitlement 차이 때문에 “Debug는 되는데 Release가 안 되는” 상황이 자주 생깁니다. 그래서 두 파일을 명확히 분리해 두는 것이 중요합니다.

6. 실제 적용 순서

  1. macos/Runner/DebugProfile.entitlements를 열고 위 최종 plist로 맞춥니다.
  2. macos/Runner/Release.entitlements를 열고 위 최종 plist로 맞춥니다.
  3. Xcode 또는 Flutter로 macOS Debug 실행을 확인합니다.
  4. 필요하면 macOS Release 빌드도 한 번 확인합니다.
만약 이 문서만 보고 새 프로젝트에 처음 적용한다면: Xcode Runner target의 Debug/Release entitlement 경로가 각각 이 두 파일을 바라보는지도 추가로 확인해야 합니다. 현재 프로젝트에서는 그 연결이 이미 되어 있어서 파일 내용만 정리하면 됩니다.

7. 확인 포인트

  • Debug 빌드에서 네트워크 요청이 막히지 않는지
  • Debug 실행 중 Flutter 엔진/JIT 관련 오류가 없는지
  • Release 빌드에서 최소한의 네트워크 요청은 정상인지
flutter run -d macos
flutter build macos

완료 기준

  • DebugProfile.entitlements에 JIT, network client, network server가 들어 있다.
  • Release.entitlements에는 network client만 들어 있다.
  • 두 파일 모두 plist 형식이 깨지지 않는다.
  • macOS Runner가 Debug/Release에서 의도한 권한으로 실행된다.
정리: 이 커밋은 작은 권한 조정처럼 보이지만, macOS Runner가 개발과 배포에서 안정적으로 동작하게 만드는 기반 설정입니다.
